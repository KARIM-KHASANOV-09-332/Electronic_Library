## 🏗 Архитектура и запуск проекта
Проект разделен на изолированные микросервисы. Все компоненты запускаются строго через Docker. Никаких локальных установок баз данных на хост-машину!

### Быстрый старт:
1. Склонируйте репозиторий: `git clone <url>`
2. Создайте файл `.env` в корне проекта (скопируйте из `.env.example` и заполните свои данные).
3. Поднимите контейнеры: `docker-compose up --build -d`
4. Выполните миграции БД (команда зависит от фреймворка).

## 🌿 Git Workflow и правила коммитов
Мы используем подход Feature Branching. В ветку `main` попадает только рабочий, протестированный код.

### 1. Правила именования веток:
* `feature/название` — для нового функционала (пример: `feature/user-auth`).
* `bugfix/название` — для исправления ошибок (пример: `bugfix/db-connection-error`).
* `hotfix/название` — для срочных исправлений в `main`.
* `docs/название` — для работы с документацией.

### 2. Спецификация Conventional Commits:
Сообщения коммитов должны быть на **английском языке** и иметь четкую структуру, чтобы легко читать историю изменений.
* `feat: add user profile page` (Новый функционал)
* `fix: resolve issue with book search` (Исправление бага)
* `refactor: simplify database queries` (Переписывание кода без изменения логики)
* `style: format python files with black` (Форматирование, пробелы, отступы)
* `docs: update readme` (Документация)

### 3. Слияние (Code Review):
* Прямые пуши в `main` **запрещены**.
* Создавайте Pull Request (PR) на GitHub.
* PR должен быть проверен и одобрен минимум одним другим разработчиком (Каримом или Мансуром) перед слиянием (Approve).

## 💻 Code Style и стандарты разработки

### Общие правила:
* Язык кода, переменных, функций и комментариев — **строго английский**.
* Названия должны отвечать на вопрос "Что это?" (существительные для переменных: `book_list`, `user_id`) или "Что делает?" (глаголы для функций: `get_books()`, `create_user()`).
* Избегайте "магических чисел" (например, `if status == 2:`). Выносите их в константы: `if status == STATUS_ACTIVE:`.

### Бэкенд и API:
* **Стиль:** Строгое соблюдение стандарта PEP 8 (для Python).
* **Типизация:** Обязательное использование Type Hints во всех функциях `def get_book(book_id: int) -> dict:`.
* **REST API:**
  * Эндпоинты должны быть существительными во множественном числе: `/api/books/`, `/api/users/`.
  * Используйте правильные HTTP-методы: `GET` (получить), `POST` (создать), `PUT`/`PATCH` (обновить), `DELETE` (удалить).
  * Возвращайте правильные HTTP-статусы (200 OK, 201 Created, 400 Bad Request, 404 Not Found).

### База Данных:
* Структура БД меняется **только** через файлы миграций. Ручные изменения таблиц через консоль СУБД запрещены (иначе у других разработчиков сломается проект).
* Названия таблиц — во множественном числе (`books`, `users`), столбцов — в единственном (`title`, `author_id`).

## 🔐 Безопасность
* **НИКОГДА** не коммитьте пароли, секретные ключи, токены API или доступы к БД.
* Все секреты должны храниться в локальном файле `.env`, который добавлен в `.gitignore`.
* В репозитории должен лежать файл `.env.example` с пустыми значениями-заглушками, чтобы команда понимала, какие переменные нужны для запуска.
